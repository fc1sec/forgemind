# ForgeMind v1.2.1: Ethical Onboarding Framework

**Date:** May 13, 2026  
**Purpose:** Define best-practice onboarding for ForgeMind users, grounded in UX research and AI ethics standards  
**Scope:** First-time user journey, agent presentation, transparent capabilities/limitations disclosure

---

## Research Foundation

This framework is grounded in:

1. **UX Research**: Appcues five-play onboarding model (progressive disclosure, personalization, contextual guidance)
2. **AI Ethics Standards**: EU AI Act Article 50 (transparency obligations), OECD AI Principles (explainability)
3. **Disclosure Limitations**: arxiv.org research on transparency fallacies and practical constraints
4. **US Regulatory Requirements**: Summary disclosure of AI system capabilities and training data sources

---

## Core Problem: ForgeMind's Complexity

**Challenge:** ForgeMind is powerful but non-obvious. New users face:
- Unclear what ForgeMind *is* (not a code generator, not a decision-maker)
- Unclear what ForgeMind *does* (structures work before execution)
- Unclear what ForgeMind *cannot* do (domain-specific reversals require expertise)
- Unclear when to use ForgeMind (planning phase, not runtime)

**Solution:** Structured onboarding journey with five phases, each addressing one aspect of user confidence.

---

## The Five-Phase Onboarding Journey

### Phase 1: Orientation (2 minutes)
**Goal:** User understands "what is ForgeMind in one sentence."

**User Story:** "I just installed ForgeMind. What do I do now?"

**Deliverables:**
1. **Interactive CLI Prompt** (on first `forgemind init`):
   ```
   Welcome to ForgeMind v1.2.1
   
   ForgeMind helps teams PLAN projects rigorously BEFORE 
   agents or humans build them.
   
   It turns vague ideas into structured, governed work packages.
   
   Ready? (y/n):
   ```

2. **Instant Visual Context** (printed after `init`):
   ```
   ✅ ForgeMind workspace initialized
   
   Next steps:
   1. cd forgemind_projects
   2. Edit sample_ai_project.md with your project details
   3. Run: forgemind intake sample_ai_project.md
   
   Takes 5 minutes. No code needed.
   
   Questions? → forgemind --help
   ```

3. **One-Sentence Value**: Appears in README and --help:
   ```
   "Structure work before execution. Catch risky assumptions,
    unclear success criteria, and unmanaged risks BEFORE
    your agent or team starts building."
   ```

**Activation Metric** (what success looks like):
- User completes `forgemind init` without errors
- User opens sample_ai_project.md

---

### Phase 2: First Success (10 minutes)
**Goal:** User experiences value from ForgeMind with minimum effort.

**User Story:** "I want to see what ForgeMind does without deep learning."

**Deliverables:**

1. **Pre-filled Example Project** (in `forgemind_projects/sample_ai_project.md`):
   - Real example: "AI Backend Automation Agent"
   - Pre-populated sections (Objective, Context, Scope, Risks, Success Criteria)
   - Clear prompts for each section
   - No required changes—can run as-is

2. **One-Command Demo** (documented in README):
   ```bash
   forgemind intake forgemind_projects/sample_ai_project.md
   # Generates 17 analysis documents in ~5 seconds
   ```

3. **Quick Win Output** (printed after intake):
   ```
   ✅ Analysis complete
   
   Generated 17 documents in forgemind_outputs/sample-ai-project/
   
   Start here:
   1. PROJECT_CHARTER.md ← High-level objective & scope
   2. RISK_REGISTER.md ← What could go wrong & how to fix it
   3. ACCEPTANCE_CRITERIA.md ← Definition of success
   
   Takes 2 minutes to review. Then you'll understand your project better.
   ```

4. **Visual Feedback** (use Rich for colors, progress):
   - Green checkmarks for successful operations
   - Blue information boxes for guidance
   - Yellow warnings for limitations
   - Red only for actual errors

**Activation Metric**:
- User runs `forgemind intake` successfully
- User opens at least one generated output file
- User understands "ForgeMind creates 17 analysis documents"

### Phase 3: Transparent Limitations Disclosure (Reading, asynchronous)
**Goal:** User explicitly understands what ForgeMind IS and IS NOT.

**User Story:** "Before I trust ForgeMind with my project, I need to know what it can and cannot do."

**Deliverables:**

1. **Honest Capabilities/Limitations Statement** (in README, linked from init):
   ```markdown
   ## What ForgeMind IS
   ✅ A readiness aid that structures work BEFORE execution
   ✅ A generator of checklists, registers, and governance documents
   ✅ A framework for surfacing risks and assumptions
   ✅ A tool for documenting decisions
   ✅ A handoff preparation system for agents or teams
   
   ## What ForgeMind IS NOT
   ❌ A code generator (doesn't write code)
   ❌ A decision-maker (you make the calls)
   ❌ A domain expert (requires your expertise in your field)
   ❌ A compliance certifier (helps document, doesn't guarantee compliance)
   ❌ A runtime monitor (doesn't watch agents work)
   ❌ A guarantee of safety (you must implement mitigations)
   ❌ A replacement for human judgment
   
   ## Why This Matters
   ForgeMind's strength is STRUCTURE, not expertise. If you need domain expertise
   (reverse engineering strategies, risk assessment in your specific field),
   you provide that. ForgeMind structures how you document and share it.
   ```

2. **Clear Domain Boundaries** (in docs/):
   Create `docs/SUPPORTED_DOMAINS.md`:
   ```markdown
   # ForgeMind Domain Support
   
   ### Fully Supported
   ✅ Software Projects (21 templates, 90%+ coverage)
   ✅ AI/ML Projects (18 templates, 85%+ coverage)
   ✅ ISO 9001 QMS (14 templates, 95%+ coverage)
   
   ### Partial Support
   ⚠️ Government Tender/Compliance (6 templates, 50% coverage)
   ⚠️ Hardware Projects (4 templates, 30% coverage)
   
   ### Not Yet Supported
   ❌ Biomedical Device (FDA regulations require specialist)
   ❌ Nuclear Systems (safety-critical, escalate to regulatory body)
   ❌ Defense/Intelligence (classified, use internal systems)
   
   ### How to Add Support
   See CONTRIBUTING_REVERSE_PATTERNS.md
   ```

3. **AI System Transparency Card** (per EU AI Act Article 50):
   Create `docs/AI_SYSTEM_TRANSPARENCY.md`:
   ```markdown
   # ForgeMind v1.2.1 Transparency Disclosure
   
   Per EU AI Act Article 50 and OECD AI Principles, here is what you should 
   know before using ForgeMind:
   
   ## What ForgeMind Does
   ForgeMind is a **local-running text analysis tool** that:
   - Reads your project description (Markdown file)
   - Applies 17 analysis templates (RDMAICSI framework)
   - Generates structured output documents
   - Requires NO internet connection after installation
   - Stores NO data on remote servers
   
   ## How ForgeMind Works
   - **Input**: Your project description (what you provide)
   - **Process**: Template-based keyword matching + structure generation
   - **Output**: 17 Markdown documents (governance, risk, acceptance criteria)
   - **Method**: Deterministic rules, NOT machine learning
   - **Data Handling**: Everything stays on your computer
   
   ## ForgeMind's Limitations
   
   ### Scope Limitation
   ForgeMind knows about TOOLS (RDMAICSI, Lean, Six Sigma), not your domain.
   - ✅ ForgeMind will help you structure a software project risk register
   - ❌ ForgeMind cannot tell you which software architectures are secure
   - You provide domain expertise; ForgeMind provides structure
   
   ### Accuracy Limitation
   ForgeMind uses keyword matching, not understanding.
   - ForgeMind will flag "database migration" as a risk IF you mention it
   - ForgeMind will NOT invent risks you didn't mention
   - ForgeMind's outputs are starting points, not final decisions
   
   ### Reversibility Limitation (v1.2 onward)
   ForgeMind provides reverse patterns for supported domains only.
   - ✅ ISO 9001: Document approval reversals documented
   - ✅ Software: Git revert + deployment strategies provided
   - ❌ Tender Procurement: Reversals constrained by compliance, escalate
   - ❌ Biomedical: Regulatory constraints prevent reversal documentation
   
   ## What ForgeMind Requires from You
   
   1. **Honest Project Description**: ForgeMind analyzes what you write.
      Vague input = vague output.
   
   2. **Domain Expertise**: You know your field. ForgeMind structures your knowledge.
   
   3. **Human Review**: Every ForgeMind output must be reviewed by someone who 
      understands your domain before execution.
   
   4. **Implementation Responsibility**: ForgeMind documents decisions. 
      You implement them and own the outcomes.
   
   ## When to Escalate (NOT use ForgeMind)
   
   - **Regulated Industries**: If compliance is required (FDA, FAA, etc.), 
     validate ForgeMind's output against regulatory requirements. 
     ForgeMind is a tool, not a regulatory body.
   
   - **Safety-Critical Systems**: Nuclear, medical devices, aircraft.
     ForgeMind helps document, but specialists must validate.
   
   - **Unknown Unknowns**: If you're unsure about risks in your domain,
     ForgeMind will only surface what you tell it. Consult domain experts first.
   
   ## Data Privacy
   
   - ✅ ForgeMind runs locally. No cloud sync.
   - ✅ Your project files stay on your computer.
   - ✅ No telemetry or usage tracking (v1.2.1).
   - ✅ You own all generated output documents.
   
   ## Support & Feedback
   
   If you find ForgeMind unhelpful, inconsistent, or misleading:
   1. Open an issue on GitHub
   2. Describe what you expected vs. what you got
   3. Include your project file (redact sensitive info)
   4. We'll investigate and update if needed
   
   Contact: fc1sec@hotmail.com (security issues only, 48-hour response)
   
   ---
   
   **Philosophy**: ForgeMind is a thinking tool, not a substitute for thinking.
   Use it to structure rigor. Bring your expertise to validate.
   ```

**Activation Metric**:
- User reads "What ForgeMind IS NOT" section
- User understands domain support level for their project
- User explicitly acknowledges need for human review

### Phase 4: Personalized Guidance (Interactive, during use)
**Goal:** Guide the user to their specific success (software vs. QMS vs. AI/ML).

**User Story:** "I'm building an AI agent. Show me what ForgeMind does for AI projects."

**Deliverables:**

1. **Role-Based Onboarding** (interactive prompt on first `intake`):
   ```bash
   $ forgemind intake my_project.md
   
   ForgeMind detected a software project. Is this correct? (y/n)
   
   Detected keywords: "API", "deployment", "rollback", "testing"
   
   This impacts your generated outputs:
   - Software-specific risk templates
   - Blue-green deployment patterns
   - Git-based reversals
   - Code review gates
   ```

2. **Domain-Specific Example Projects** (in examples/):
   ```
   examples/
   ├── ai_agent_project.md          ← Best for: ML teams
   ├── software_feature_project.md  ← Best for: Backend/frontend
   ├── qms_sgc_project.md           ← Best for: Compliance teams
   ├── tender_analysis_project.md   ← Best for: Government/bids
   └── operational_process.md       ← Best for: Process improvement
   ```
   Each example pre-filled for that domain with realistic structure.

3. **Contextual Help** (via `--explain` flag):
   ```bash
   $ forgemind intake --explain my_project.md
   
   Explains:
   - Why ForgeMind asks for these sections
   - What each generated document is used for
   - Which documents are critical for your domain
   - Where to get domain expertise if ForgeMind detects gaps
   ```

4. **Post-Intake Guidance** (printed after analysis):
   For AI projects:
   ```
   ✅ Generated 17 documents for AI project
   
   For AI Projects, prioritize:
   1. RISK_REGISTER.md
      - Model hallucination risk
      - Unsafe autonomy scope
      - Data poisoning risk
   
   2. TOOL_PERMISSION_MATRIX.md
      - What your AI agent CAN access
      - What it CANNOT access
      - Approval gates for risky operations
   
   3. AGENT_HANDOFF.md
      - Clear boundaries for agent autonomy
      - Human review gates (critical)
      - Rollback procedures
   
   Next: Review these 3 docs with your team before agent development starts.
   ```

**Activation Metric**:
- User sees domain-appropriate guidance
- User understands which outputs matter most for their context
- User knows when to escalate (e.g., to AI safety specialist)

### Phase 5: Ongoing Value Reinforcement (After first success)
**Goal:** Show user that ForgeMind improved their outcomes.

**User Story:** "We used ForgeMind on 3 projects. Here's what it helped us catch."

**Deliverables:**

1. **Post-Execution Feedback Loop** (v1.3+):
   After project completion, prompt:
   ```bash
   $ forgemind feedback project_name
   
   How did this project go?
   1. Great—exceeded expectations
   2. Good—met expectations
   3. Okay—met some expectations
   4. Poor—had significant issues
   
   [Follow-up: What could ForgeMind have warned about earlier?]
   
   [Stores anonymized data to improve future recommendations]
   ```

2. **Lessons Dashboard** (v1.3+):
   ```bash
   $ forgemind dashboard
   
   Your ForgeMind History
   ✅ 3 projects using ForgeMind
   ✅ 8 risks identified across projects
   ✅ 5 assumptions validated before execution
   ⚠️  2 projects caught scope creep early
   
   Most Common Risks (your projects):
   - Schedule pressure (3/3)
   - Scope creep (2/3)
   - Unclear acceptance criteria (3/3)
   
   → ForgeMind helped catch these early. Good sign!
   ```

3. **Version Safety Reminders** (v1.2.1+):
   Automatic non-blocking notification:
   ```
   ✓ Update available: v1.2.1 → v1.3.0
   
   What's new:
   - Knowledge graph for learning from projects
   - Recommendation engine (personalized suggestions)
   - Feedback loop (help us improve)
   
   Safe to update: All v1.x projects fully compatible.
   Update: pip install --upgrade forgemind
   ```

**Activation Metric**:
- User completes at least 2 projects with ForgeMind
- User runs `forgemind feedback` on completed projects
- User sees ForgeMind improvements over time

## Ethical Principles (Grounded in Research)

### 1. Transparency Over Surprise
- **Principle**: Users should know what ForgeMind does and doesn't do *before* relying on it.
- **Implementation**: Transparency card in README. Domain support clearly stated. Limitations documented.
- **Standard**: EU AI Act Article 50, OECD AI Principles (Explainability)

### 2. Competence Over Overreach
- **Principle**: ForgeMind is a tool for *structure*, not *expertise*. Don't pretend to know domains we don't.
- **Implementation**: Plugin system for domain expertise. Escalation when outside ForgeMind's lane.
- **Standard**: OECD AI Principles (Robustness, Security, Accountability)

### 3. Progressive Disclosure Over Feature Dump
- **Principle**: Teach users what matters most first. Advanced concepts come later.
- **Implementation**: Five-phase journey (Orientation → First Success → Limitations → Personalization → Ongoing).
- **Standard**: UX Research (Appcues, NN Group)

### 4. Honesty Over Hype
- **Principle**: Acknowledge what ForgeMind gets wrong and what it can't do.
- **Implementation**: "What ForgeMind IS NOT" section. Transparent confidence scoring.
- **Standard**: arxiv Research on Transparency Fallacies (acknowledge limits of disclosure)

### 5. User Agency Over Automation
- **Principle**: ForgeMind documents human decisions. It doesn't make them.
- **Implementation**: Every output is labeled as "readiness aid, not guarantee." Human review gates required.
- **Standard**: OECD AI Principles (Human Agency and Oversight)
